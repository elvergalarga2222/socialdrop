# PR #32 — 🔴 Workspaces: multi-cuenta real y modo agencia

**Área:** Feature / Seguridad
**Prioridad:** 🔴 Crítico (redefine la clave de tenant antes de ejecutar #12/#13/#14 — ejecutar su schema JUNTO con esos PRs, no después)
**Esfuerzo estimado:** XL (3+ días)
**Depende de:** PR #12 (guard/@CurrentUser), PR #31 (registro). Modifica el diseño de #13, #14, #28 y deja un punto de decisión en #29.

**Problema:**
El diseño actual (PRs #12-#14) usa `userId` como clave de tenant. Eso no modela el caso agencia: un mismo login gestionando varios clientes, cada uno con sus propias cuentas sociales, posts, calendario, stats, brain y competidores. Además, la percepción de "límite de cuentas conectadas" no viene de un cap propio (verificado: no existe ningún `MAX_ACCOUNTS`/`length >= 5` en schema, controller ni UI) sino de que FB/IG se conectan con credenciales estáticas de env (`FACEBOOK_PAGE_ID`/`INSTAGRAM_ACCOUNT_ID`) → 1 cuenta por instalación. El desbloqueo de eso es PR #14 (OAuth real por `profileId`); este PR añade la capa que permite N cuentas organizadas por cliente.

**Solución propuesta:**

1. **Modelo de datos:**
   ```prisma
   model Workspace {
     id        String   @id @default(uuid())
     name      String
     createdAt DateTime @default(now())
     updatedAt DateTime @updatedAt
     members      WorkspaceMember[]
     integrations Integration[]
     // + relaciones con todas las tablas tenant (ver punto 3)
   }

   model WorkspaceMember {
     id          String        @id @default(uuid())
     workspaceId String
     workspace   Workspace     @relation(fields: [workspaceId], references: [id], onDelete: Cascade)
     userId      String
     user        User          @relation(fields: [userId], references: [id], onDelete: Cascade)
     role        WorkspaceRole // OWNER | MEMBER
     createdAt   DateTime      @default(now())
     @@unique([workspaceId, userId])
     @@index([userId])
   }
   ```
   Regla: todo workspace tiene **exactamente un OWNER** (invariante de servicio; simplifica billing y borrado). `User.lastActiveWorkspaceId String?` para restaurar el contexto al entrar.
2. **Workspace default transparente:** al registrarse (hook en PR #31, misma transacción) se crea `Workspace{name: name ?? "Mi espacio"}` + membership OWNER. Un usuario con **un solo** workspace nunca ve el concepto: el switcher no se muestra y todo funciona como hoy.
3. **`workspaceId` como clave de tenant:** `Integration.userId` → `Integration.workspaceId` (unique pasa a `[workspaceId, platform, profileId]`) y todas las tablas tenant listadas en PR #14 (`Post`, `ContentItem`, `Flow`, `Contact`, `Competitor`, `GeneratedScript`, `PlatformMetrics`, `PostAnalytics`, `QueueSlot`, `YoutubeComment`, `YoutubeAutoReply`, `BrandProfile`, `ContentStrategy`, `GrowthGoal`, `ContentBrain`, `DriveConfig`) migran de `userId` a `workspaceId`, con FK + `onDelete: Cascade` hacia `Workspace` e índices `@@index([workspaceId, ...])`. `User` conserva solo lo personal (credenciales, `Subscription` si se decide así en #29).
4. **Migración/backfill:** por cada `User` existente, crear su workspace default + membership OWNER y mapear `userId → workspaceId` en todas las tablas (incluye `demo-user` → "Workspace Demo"). Es la MISMA migración de backfill que ya planeaba PR #14, con el paso extra de crear el workspace; por eso este PR debe ejecutarse junto a #14, no encima.
5. **Contexto activo (cómo se deriva `workspaceId`):**
   - El **JWT no lleva workspace** (si lo llevara, cambiar de cliente exigiría re-emitir token y una membership revocada seguiría viva hasta expirar).
   - El frontend envía `X-Workspace-Id` en cada request; un `WorkspaceGuard` (encadenado tras el `AuthGuard` de PR #12) valida `WorkspaceMember{ userId: token.sub, workspaceId: header }` y expone `@ActiveWorkspace()` a los controllers. Header ausente/ajeno → 403.
   - Al cambiar en el switcher se persiste `lastActiveWorkspaceId`.
6. **UI:** switcher de workspace en el `AppShell` (encima de la nav), visible solo con ≥2 memberships; página `settings/workspaces` para crear workspace (modo agencia), renombrar, e invitar/quitar miembros (`POST /api/workspaces/:id/members { email, role }` — solo OWNER; si el email no tiene cuenta, queda invitación pendiente que se materializa al registrarse). `posts/new`, calendario, queue, analytics, brain y competitors no cambian de UX: solo consultan con el workspace activo.
7. **Sin límite artificial de cuentas:** no se introduce ningún cap propio; el número de `Integration` por workspace lo limita únicamente el plan (punto 8). Restricciones externas de Meta documentadas aparte (no son bugs): cuenta IG debe ser Business/Creator ligada a página FB; rate limit ~25 publicaciones/24h **por cuenta IG**; gestionar cuentas de terceros requiere los scopes + App Review de PR #30.
8. **Límites por plan (conexión con #28/#29):** el plan define `maxWorkspaces` y `maxAccountsPerWorkspace` (config, no hardcode). El enforcement vive en `workspaces.create` y en `integrations.connect`. `UsageCounter` (PR #28) pasa a keyearse por `workspaceId`, con agregación por pagador para topes de plan.

**Cambios sobre PRs existentes (modificar, no duplicar):**
- **PR #12:** los controllers no reemplazan `@Query('userId')` por `user.sub` a secas: los recursos tenant usan `@ActiveWorkspace()` (este PR); solo lo personal (perfil, billing) usa `@CurrentUser()`.
- **PR #13:** el criterio de isolación cambia a `where: { id, workspaceId }` con `workspaceId` derivado del contexto activo validado. Tests: cross-user (403/404) **y** cross-workspace del mismo user (un OWNER de A no lee recursos de B pasando el header de B si no es miembro; y con dos workspaces propios, los datos no se mezclan).
- **PR #14:** las FK/índices/backfill apuntan a `Workspace` en vez de `User`; el trabajo de providers por `profileId` queda igual y es prerequisito del multi-cuenta.
- **PR #28:** cuotas por `workspaceId` (ver punto 8).
- **PR #29:** ✅ **decisión tomada (2026-07): Opción A** — `Subscription` cuelga de `User` con `quantity` en el subscription item = nº de workspaces contratados. `maxWorkspaces = subscription.quantity`; crear un workspace por encima dispara upgrade prorrateado de quantity (o bloquea, según config). Este PR añade `Workspace.frozenAt` para el congelamiento por downgrade. Detalle completo en `docs/prs/PR-29.md`.

**Criterios de aceptación:**
- [ ] Registro crea workspace default de forma transparente; con 1 workspace el switcher no aparece
- [ ] Un user con 2+ workspaces cambia de cliente desde el AppShell y TODOS los datos (posts, calendario, stats, brain, competitors, integraciones) cambian de contexto
- [ ] `X-Workspace-Id` de un workspace del que no eres miembro → 403; membership revocada corta el acceso al instante
- [ ] N cuentas conectadas por workspace sin cap propio (solo límite de plan)
- [ ] Backfill: datos existentes quedan en el workspace default de su usuario, sin pérdida
- [ ] Tests cross-user y cross-workspace en cada módulo tenant

**Notas técnicas:**

Invitaciones con email real dependen del `MailProvider` de PR #31 (en dev, consola). El borrado de workspace es solo-OWNER y con confirmación (cascade destruye todo el contenido del cliente).
