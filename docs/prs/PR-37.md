# PR #37 — Animación de entrada en gráficas del dashboard (delta sobre PR #01-#04)

**Área:** Diseño / Frontend
**Prioridad:** 🟢 Mejora
**Esfuerzo estimado:** M (2-8h)
**Depende de:** PR #33 (tokens/series), PR #34 (motion), PR #01-#04 (KPIs y gráficas)

**Problema:**
Los PRs #01-#04 especifican las gráficas del dashboard (KPIs con deltas, barras por plataforma, heatmap de mejores horas, histórico por post) pero sin dirección de motion ni sistema de color de series. Sin este delta, cada gráfica se implementaría con colores y entradas ad-hoc.

**Solución propuesta (este PR MODIFICA #01-#04, no duplica su spec):**

1. **Delta sobre PR #01 (fila de KPIs):**
   - Cifra con **count-up** al montar: `animate(0 → valor, 600ms, --ease-out)` sobre `MotionValue`, renderizada en JetBrains Mono `tabular-nums` (el ancho no baila). Con `prefers-reduced-motion`: valor final directo, sin animación.
   - Delta ▲/▼ entra 150ms después del count-up (opacity). Tarjetas en secuencia vía `Stagger` (PR #34) — este es el "momento orquestado" del dashboard: KPIs → gráficas, nunca todo a la vez.
2. **Delta sobre PR #02 (barras apiladas por plataforma):**
   - Entrada: barras crecen desde la línea base (`scaleY 0→1`, `transformOrigin: bottom`, 400ms, stagger 40ms por barra).
   - Colores de segmento = `--color-chart-1..4` en orden fijo por plataforma (la identidad de plataforma NO se repinta al filtrar); gap de 2px color superficie entre segmentos apilados y entre barras adyacentes.
3. **Delta sobre PR #03 (heatmap 7×24 de mejores horas):**
   - Entrada: fade de celdas con stagger 8ms por columna (barrido izquierda→derecha), 250ms total aprox.
   - El heatmap es **secuencial de un solo tono** (violeta de marca, claro→oscuro), nunca multicolor; valor numérico visible en las celdas top-3 (mono 11).
4. **Delta sobre PR #04 (histórico de métricas por post — línea/área):**
   - Entrada: trazo de línea con `pathLength 0→1` (500ms, `--ease-out`) + fade del relleno de área al completarse; una sola vez al montar.
   - Actualizaciones por cambio de filtro/período: **sin re-animar la entrada** — transición de 200ms entre estados de la misma marca.
5. **Reglas comunes** (heredadas de PR #33 + sistema dataviz, aplican a las 4):
   - Labels, ejes y leyendas en tokens de tinta (nunca el color de la serie); grid recesivo `rgba(255,255,255,0.04)`.
   - ≥2 series ⇒ leyenda presente + etiquetas directas selectivas; 1 serie ⇒ el título nombra la serie, sin leyenda.
   - Hover/tooltip son parte de la spec de #02-#04 y NO se animan con entrada (respuesta inmediata, 0ms de delay).
   - Un solo eje Y por gráfica (nunca dual-axis).

**Archivos a modificar:**
- Los componentes de gráfica que introduce la implementación de PR #01-#04 (`apps/web/src/components/charts/*` según los cree el ejecutor)
- `apps/web/src/app/page.tsx` (orquestación Stagger de la fila de KPIs)

**Criterios de aceptación:**
- [ ] KPIs con count-up en mono tabular; secuencia de entrada visible; reduced-motion ⇒ valores finales sin animación
- [ ] Barras crecen desde la base con stagger; línea se traza con pathLength; heatmap barre por columnas
- [ ] Cambios de filtro transicionan (200ms) sin repetir la animación de entrada
- [ ] Series con `--color-chart-*` en orden fijo; labels en tinta; gaps de 2px entre marcas
- [ ] Tooltips instantáneos (sin delay de entrada)

**Notas técnicas:**
Ejecutar idealmente en la misma tanda que #01-#04 (el ejecutor implementa gráfica + motion de una vez usando este delta como parte de la spec). `pathLength` requiere que las líneas sean SVG `<path>` — si #02-#04 se implementan con Recharts, usar sus props de animación equivalentes con las mismas duraciones/easings; si son SVG propios, motion directo.
