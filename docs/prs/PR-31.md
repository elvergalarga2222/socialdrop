# PR #31 — 🔴 Auth real: registro, verificación de email y reset de contraseña

**Área:** Seguridad / Feature
**Prioridad:** 🔴 Crítico (bloqueante de lanzamiento, junto con PR #12)
**Esfuerzo estimado:** L (1-2 días)
**Depende de / se ejecuta con:** PR #12 (multi-tenancy). Ver "Cambios sobre PRs existentes" abajo.

**Problema:**
PR #12 define la infraestructura (JWT `sub = user.id`, `AuthGuard` global, `@CurrentUser()`, quitar el AUTH BYPASS) pero deja el registro/login como un esbozo. Para lanzar hacen falta los flujos completos de cuenta: signup, verificación de email y recuperación de contraseña. Hoy el "login" es una `APP_PASSWORD` compartida y `User.passwordHash` existe pero nunca se escribe.

**Solución propuesta:**

1. **Registro** — `POST /api/auth/register { email, password, name? }`:
   - Hash con **argon2id** (preferido; bcrypt cost≥12 como alternativa si hay fricción de dependencias en el Dockerfile alpine).
   - Validación: email RFC + password ≥8 chars. Email único (409 si existe, con mensaje neutro para no filtrar existencia — devolver el mismo "revisa tu correo").
   - Crea `User` + dispara el hook de workspace default (PR #32) en la misma transacción.
   - Emite token de verificación y "envía" email (ver punto 3).
2. **Login** — `POST /api/auth/login { email, password }` → JWT `{ sub: user.id, tokenVersion }` (idéntico contrato que PR #12), cookie httpOnly + expiración de `JWT_EXPIRY`. Comparación con verificación de hash en tiempo constante. Opcional: bloquear login si `emailVerifiedAt` es null tras N días de gracia (config `REQUIRE_EMAIL_VERIFICATION`, default off para no frenar la beta).
3. **Verificación de email** (diseño completo; el *envío* real puede diferirse):
   - Campo `User.emailVerifiedAt DateTime?`.
   - Tabla `AuthToken { id, userId FK, tokenHash, type: VERIFY_EMAIL | RESET_PASSWORD, expiresAt, usedAt?, createdAt }` — se genera token aleatorio de 32 bytes, se guarda **solo su SHA-256**, single-use.
   - `POST /api/auth/verify-email { token }` → marca `emailVerifiedAt`, invalida el token.
   - **Envío tras interfaz** `MailProvider { send(to, template, vars) }` con implementación `ConsoleMailProvider` (log en dev) y hueco para Resend/SMTP en un PR posterior. Así el flujo queda funcional end-to-end en dev sin bloquear por el proveedor de email.
4. **Olvidé mi contraseña:**
   - `POST /api/auth/request-password-reset { email }` → siempre responde 200 (no filtrar existencia); si el email existe, crea `AuthToken` tipo RESET_PASSWORD con **expiración 60 min**.
   - `POST /api/auth/reset-password { token, newPassword }` → valida hash+expiración+single-use, actualiza `passwordHash`, incrementa `User.tokenVersion` (invalida **todos** los JWT emitidos antes del reset — el `AuthGuard` de PR #12 compara `tokenVersion` del JWT contra el de BD).
5. **Rate limiting** en `register`, `login` y `request-password-reset` (por IP y por email, p. ej. `@nestjs/throttler` + Redis): 5/min y 20/h. Sin esto, el reset es un oráculo de spam y el login es fuerza-bruteable.
6. **UI:** páginas `/login` (rehacer la actual de APP_PASSWORD), `/register`, `/forgot-password`, `/reset-password?token=`. Quitar el AUTH BYPASS del middleware (ya especificado en PR #12).

**Cambios sobre PRs existentes (modificar, no duplicar):**
- **PR #12:** su punto 1 ("Sistema de registro/login real…") se **traslada íntegro a este PR**. PR #12 queda acotado a: `AuthGuard` global + `tokenVersion` check, `@CurrentUser()`, eliminar `userId` de query params en todos los controllers, quitar el AUTH BYPASS del middleware Next, y migrar los `@default("demo-user")`. Orden de ejecución recomendado: #12 y #31 en la misma tanda (el guard sin flujos de cuenta no es usable en producción, y viceversa).
- **PR #13:** sin cambios por este PR (los cambios a #13 vienen de PR #32).

**Archivos a modificar:**
- `libs/prisma/prisma/schema.prisma` (`User.emailVerifiedAt`, `User.tokenVersion Int @default(0)`, tabla `AuthToken` + migración)
- `apps/api/src/modules/auth/*` (service, controller, dto, mail-provider, throttling)
- `apps/web/src/app/login/page.tsx`, nuevos `register/`, `forgot-password/`, `reset-password/`
- `apps/web/src/middleware.ts`, `apps/web/src/lib/api.ts` (coordinado con PR #12)

**Criterios de aceptación:**
- [ ] Registro → usuario con argon2id/bcrypt en `passwordHash` + workspace default (PR #32) en una transacción
- [ ] Login devuelve JWT `sub = user.id`; credenciales inválidas → 401 genérico
- [ ] Token de verificación/reset: hasheado en BD, single-use, expira (24h verify / 1h reset)
- [ ] Reset de contraseña invalida los JWT anteriores (`tokenVersion`)
- [ ] `request-password-reset` responde 200 exista o no el email
- [ ] Rate limiting activo en los 3 endpoints sensibles
- [ ] Flujo de email operativo en dev vía `ConsoleMailProvider` (envío real diferido a otro PR)

**Notas técnicas:**
No usar el `SESSION_SECRET`/HMAC actual del middleware Next (se elimina con el bypass). El JWT vive en cookie httpOnly + SameSite=Lax; `apiFetch` pasa a `credentials: 'include'`. Los mensajes de error de auth son deliberadamente genéricos (no filtrar si un email está registrado).
