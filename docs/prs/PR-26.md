# PR #26 — 🔴 Storage duradero en Cloudflare R2

**Área:** Infraestructura
**Prioridad:** 🔴 Crítico (antes de escalar usuarios)
**Esfuerzo estimado:** L (1-2 días)

**Problema:**
La media vive en el disco local del VPS (`/app/uploads`, servida estática para que Meta la descargue). El borrado a los 5 min post-publicación es una mitigación frágil: si un post reintenta o Meta tarda en descargar, el archivo puede desaparecer antes de tiempo; y si el disco se llena, **las subidas y publicaciones fallan**. Sin durabilidad: si el VPS muere, se pierde la media pendiente.

**Solución propuesta:**
1. Subir media a R2 (S3-compatible) con presigned PUT desde el cliente (`uploadFileXHR` → presigned URL).
2. Servir URLs públicas HTTPS de R2 a Meta/TikTok (requisito de `video_url`/`image_url`).
3. Retirar el borrado agresivo del disco; usar lifecycle rules de R2 para media temporal (ej. 30 días).
4. Mantener compat con archivos locales existentes durante la transición.

**Archivos a modificar:**
- `apps/api/src/modules/media/*`, `apps/api/src/modules/bulk/*`
- `apps/web/src/lib/uploadMedia.ts`
- `apps/api/src/modules/posts/processors/post-scheduler.processor.ts` (delete-media)
- `docker-compose.yml` / env (credenciales R2)

**Criterios de aceptación:**
- [ ] La media se sirve desde R2 con URL HTTPS pública que Meta puede descargar
- [ ] Publicar no depende del disco del VPS
- [ ] Lifecycle de limpieza configurado en R2
