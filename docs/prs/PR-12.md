# PR #12 — 🔴 Multi-tenancy: userId desde el token, no desde query param

**Área:** Seguridad / Feature
**Prioridad:** 🔴 Crítico — bloqueante #1 del lanzamiento
**Esfuerzo estimado:** XL (3+ días)

**Problema:**
Todos los endpoints reciben `userId` como query param sin validarlo contra el usuario autenticado → cualquiera puede pasar `?userId=<otro>` y leer/escribir datos ajenos (IDOR total). El frontend usa `'demo-user'` hardcodeado. El `AuthGuard` existe pero `apps/web/src/middleware.ts` tiene un `AUTH BYPASS — return NextResponse.next()` en la primera línea, y el login es una sola `APP_PASSWORD` compartida con JWT `sub:'user'` (si `APP_PASSWORD` no está configurada, el auth queda deshabilitado).

**Solución propuesta:**
1. Registro/login real (email+password → `User.passwordHash` ya existe; bcrypt/argon2). JWT con `sub = user.id`.
2. `AuthGuard` global; decorador `@CurrentUser()` que extrae `req.user.sub`; **derivar `userId` del token** en todos los controllers, ignorando cualquier `userId` del query.
3. Quitar el AUTH BYPASS del middleware de Next; token en cookie httpOnly.
4. Migrar `@default("demo-user")` del schema → `userId` explícito requerido; script de migración de los datos demo existentes.

**Archivos a modificar:**
- `apps/api/src/modules/auth/*` (register, hashing)
- Todos los `*.controller.ts` (reemplazar `@Query('userId')` por `@CurrentUser()`)
- `apps/web/src/middleware.ts`, `apps/web/src/lib/api.ts`
- `libs/prisma/prisma/schema.prisma`

**Criterios de aceptación:**
- [ ] Ningún endpoint acepta `userId` del cliente; se deriva del token
- [ ] Un usuario no puede leer/mutar datos de otro (tests de isolación)
- [ ] El middleware de Next ya no hace bypass

**Notas técnicas:**
Hacerlo por módulos con tests de isolación en cada uno. Integra naturalmente con Stripe (plan por `User`) más adelante.
