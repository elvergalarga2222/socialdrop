# PR #29 — 🔴 Suscripciones con Stripe + planes por usuario

**Área:** Feature / Infraestructura
**Prioridad:** 🔴 Crítico (para monetizar el lanzamiento; depende de PR #12)
**Esfuerzo estimado:** XL (3+ días)

**Problema:**
La matriz del Análisis 5 marca Stripe/planes como impacto alto, pero no existe nada: no hay modelo de suscripción, ni checkout, ni webhooks de Stripe, ni gating por plan. Sin esto no se puede cobrar a los usuarios que entren al lanzamiento, y las cuotas de uso (PR #28) no tienen sobre qué plan aplicarse.

**Solución propuesta:**
1. **Modelo de datos:** `Subscription { id, userId (FK User, unique), stripeCustomerId, stripeSubscriptionId, plan (FREE|STARTER|PRO), status (active|past_due|canceled|trialing), currentPeriodEnd, createdAt, updatedAt }`. Enum `Plan` con límites definidos en config (no hardcodeados en queries).
2. **Checkout:** endpoint `POST /api/billing/checkout` que crea una Stripe Checkout Session (mode=subscription) y devuelve la URL; `POST /api/billing/portal` para el Customer Portal (cambios de plan/cancelación gestionados por Stripe, no UI propia).
3. **Webhook Stripe:** `POST /api/webhooks/stripe` con verificación de firma (`stripe.webhooks.constructEvent` con raw body — mismo patrón que PR #19). Manejar `checkout.session.completed`, `customer.subscription.updated`, `customer.subscription.deleted`, `invoice.payment_failed` → actualizar `Subscription.status/plan/currentPeriodEnd`.
4. **Gating:** `PlanGuard`/decorador que lee el plan del usuario (del token, PR #12) y lo expone a los checks de cuota (PR #28). Degradar a FREE al cancelar, no borrar datos.
5. **UI mínima:** página `/settings/billing` con plan actual, botón upgrade (checkout) y botón "Gestionar suscripción" (portal).

**Planes sugeridos (validar contra la estimación de costos en docs/prs/PR-28.md):**
- FREE: 1 cuenta social, 10 posts/mes, sin análisis de competidores ni Content Brain.
- STARTER (~US$19/mes): 3 cuentas, posts ilimitados, 20 análisis de competidor/mes, 50 guiones IA/mes.
- PRO (~US$49/mes): 10 cuentas, 100 análisis/mes, guiones ilimitados (fair use), prioridad en colas.

**Archivos a modificar:**
- `libs/prisma/prisma/schema.prisma` (`Subscription`, enum `Plan` + migración)
- Nuevo `apps/api/src/modules/billing/*` (service, controller, webhook, plan.guard)
- `apps/web/src/app/settings/billing/page.tsx` (nueva)
- `apps/api/src/app/app.module.ts`, `docker-compose.yml`/env (`STRIPE_SECRET_KEY`, `STRIPE_WEBHOOK_SECRET`, price IDs)

**Criterios de aceptación:**
- [ ] Checkout → webhook → `Subscription` activa, sin intervención manual
- [ ] Firma del webhook de Stripe verificada con raw body (petición sin firma → 400)
- [ ] Cancelación/impago degrada a FREE sin borrar datos
- [ ] El plan del usuario es legible por los guards de cuota (PR #28)
- [ ] Reintento de webhook idempotente (procesar el mismo event.id dos veces no duplica)

**Notas técnicas:**
Depende de PR #12 (usuarios reales): `Subscription` cuelga de `User`. Usar Stripe Checkout + Portal (no Elements) para minimizar superficie PCI y UI. Guardar `stripeEventId` procesados para idempotencia. Los precios/límites viven en config para poder ajustarlos sin migración.
