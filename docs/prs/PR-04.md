# PR #04 — Serie temporal de métricas por post de Instagram (histórico real)

**Área:** Feature / Dashboard
**Prioridad:** 🟡 Importante
**Esfuerzo estimado:** L (1-2 días)

**Problema:**
`PostAnalytics` tiene `@@unique([userId, platform, platformPostId])` y se actualiza con `upsert` → cada sync **pisa** el valor anterior. Es imposible graficar cómo evolucionó un post (likes/comentarios/alcance/guardados/impresiones) en el tiempo. Además el `Post` interno no guarda su `media_id` de Instagram a nivel de post — solo `PostIntegration.platformPostId`, y `PostAnalytics.postId` nunca se puebla.

**Solución propuesta:**
1. **Nueva tabla append-only** `PostMetricSnapshot { id, userId, platform, platformPostId, postId?, likes, comments, shares, saves, reach, impressions, views, recordedAt }` con `@@index([platformPostId, recordedAt])`. Sin unique sobre el post → una fila por captura.
2. **Job periódico** BullMQ `metrics-snapshot` (cron cada 6h, patrón de `MetricsService.scheduledSync`) que para cada post publicado en los últimos ~30 días llame a `/{ig-media-id}/insights?metric=reach,impressions,saved` + `like_count,comments_count` e **inserte** un snapshot.
3. **Vínculo Post↔media_id:** usar `PostIntegration.platformPostId` (ya se guarda al publicar en `post-scheduler.processor.ts`) como `platformPostId`; poblar `postId` con `PostIntegration.postId`. Backfill opcional para posts ya publicados.
4. Mantener `PostAnalytics` como "último snapshot".
5. Endpoint `GET /api/metrics/post/:platformPostId/history` → array ordenado para graficar.

**Archivos a modificar:**
- `libs/prisma/prisma/schema.prisma` (+ migración)
- `apps/api/src/modules/metrics/metrics.service.ts`
- `apps/api/src/modules/metrics/metrics.processor.ts`
- `apps/api/src/modules/metrics/metrics.controller.ts`

**Criterios de aceptación:**
- [ ] Cada corrida de sync **inserta** una fila nueva (no sobrescribe)
- [ ] `insights` de IG se piden vía `/insights` (`saved`, `reach`, `impressions`), no como campos de `/media`
- [ ] Endpoint de historial devuelve la evolución ordenada por `recordedAt`
- [ ] Retención: job de limpieza que borra snapshots > 180 días

**Notas técnicas:**
Cuidado con rate limits de Graph API (ver PR #20): batchear `insights` y espaciar. `impressions` está deprecado para algunos tipos de media en versiones recientes — degradar con gracia si el campo no viene.
