# PR #10 — 🔴 Arreglar drip sequences (handler sequence-step faltante)

**Área:** Feature / Bug
**Prioridad:** 🔴 Crítico
**Esfuerzo estimado:** M (2-8h)

**Problema:**
`SequencesService.enroll` encola jobs `'sequence-step'` con delays acumulados, pero `PostSchedulerProcessor.process()` **no tiene rama para `sequence-step`**: los jobs se agregan sin `data.type`, caen en `return null` y el paso nunca envía el DM. La feature "Secuencias drip" es un no-op silencioso — parece funcionar (el job "completa") pero no hace nada.

**Solución propuesta:**
Añadir en `process()` el manejo de jobs `sequence-step` (por `job.name`, igual que `delete-media`) que envíe el DM al `contactAccountId` en la `platform` indicada usando el **token de la integración del usuario dueño** (no el env global `FACEBOOK_ACCESS_TOKEN` — ver PR #17). Registrar el resultado en `DebugLog` y en `FlowExecution`.

**Archivos a modificar:**
- `apps/api/src/modules/posts/processors/post-scheduler.processor.ts`
- (reutilizar) `apps/api/src/modules/flows/flow.engine.ts` (`sendDM`)

**Criterios de aceptación:**
- [ ] Un `sequence-step` encolado envía el mensaje al vencer su delay
- [ ] El fallo reintenta con backoff y queda registrado (no silencioso)
- [ ] Test que verifica que el job name/type se enruta correctamente
