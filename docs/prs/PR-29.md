# PR #29 — 🔴 Suscripciones con Stripe: plan por User + quantity por Workspace

**Área:** Feature / Infraestructura
**Prioridad:** 🔴 Crítico (para monetizar el lanzamiento)
**Esfuerzo estimado:** XL (3+ días)
**Depende de:** PR #12 (usuarios reales), PR #31 (registro), PR #32 (Workspaces). Los límites de uso finos viven en PR #28.

> **Decisión de producto (2026-07, confirmada por el usuario):** `Subscription` cuelga de **`User`** (Opción A). El pagador es la agencia; los clientes finales NO pagan su propio workspace. El cobro por cliente gestionado se modela con **`quantity` en el subscription item de Stripe = número de workspaces contratados** (per-seat), bajo una sola suscripción/factura. No hay suscripción a nivel de `Workspace` por ahora.

**Problema:**
No existe modelo de suscripción, checkout, webhooks de Stripe ni gating por plan. Sin esto no se puede cobrar en el lanzamiento, y las cuotas (PR #28) y el límite de workspaces (PR #32) no tienen sobre qué aplicarse.

**Solución propuesta:**

1. **Modelo de datos:**
   ```prisma
   model Subscription {
     id                   String   @id @default(uuid())
     userId               String   @unique
     user                 User     @relation(fields: [userId], references: [id], onDelete: Cascade)
     stripeCustomerId     String   @unique
     stripeSubscriptionId String?  @unique
     plan                 Plan     @default(FREE)   // FREE | STARTER | PRO
     status               SubStatus @default(ACTIVE) // ACTIVE | TRIALING | PAST_DUE | CANCELED
     quantity             Int      @default(1)      // nº de workspaces contratados (espejo del subscription item)
     currentPeriodEnd     DateTime?
     createdAt            DateTime @default(now())
     updatedAt            DateTime @updatedAt
   }
   ```
   El **tier** (STARTER/PRO) define las capacidades *por workspace* (cuentas conectadas por workspace, cuotas de IA/análisis de PR #28, features); la **quantity** define *cuántos* workspaces. Precio = precio unitario del tier × quantity (proration automática de Stripe al cambiar quantity).
2. **Planes** (unitarios por workspace/mes; config, no hardcode — calibrar contra `docs/prs/PR-28.md`):
   - **FREE:** sin suscripción de Stripe. 1 workspace, 1-2 cuentas, sin análisis de competidores ni Content Brain.
   - **STARTER (~US$19/workspace/mes):** hasta 5 cuentas por workspace, 20 análisis de competidor/mes/workspace, 50 guiones IA/mes/workspace.
   - **PRO (~US$49/workspace/mes):** hasta 15 cuentas por workspace, 100 análisis/mes/workspace, guiones ilimitados (fair use), prioridad en colas.
3. **`maxWorkspaces` derivado de la quantity contratada:** no es un número del plan, es `subscription.quantity` (FREE ⇒ 1). Fuente de verdad: Stripe; `Subscription.quantity` es un espejo sincronizado por webhook, y el enforcement lee el espejo (no llama a Stripe en el hot path).
4. **Crear workspace por encima de lo contratado** (`workspaces.create` en PR #32 consulta este módulo):
   - Si `ownedWorkspaces < quantity` → crear normal.
   - Si `ownedWorkspaces >= quantity` → **no se crea todavía**. La API responde `402 { code: 'workspace_limit', current, contracted, unitPrice }` y la UI muestra un modal de upsell con el prorrateo estimado: "Añadir 1 workspace más (+$X/mes)".
   - Aceptar en el modal llama a `POST /api/billing/quantity { quantity: n+1 }` → actualiza el subscription item en Stripe (`proration_behavior: 'create_prorations'`) → el webhook `customer.subscription.updated` sincroniza `Subscription.quantity` → el cliente reintenta la creación (la UI encadena esto automáticamente: upgrade → poll/refetch → create).
   - Config `BILLING_AUTO_QUANTITY_UPGRADE` (default `true`): en `false`, el modal no ofrece el upgrade en un click sino que enlaza a `/settings/billing` y la creación queda bloqueada hasta actualizar el plan (modo conservador que pidió tener disponible).
   - Usuario FREE en el límite → el modal ofrece el checkout de STARTER/PRO (no un cambio de quantity).
5. **Reducción de quantity / cancelación (regla "congelar, no borrar"):**
   - Bajar quantity por debajo de `ownedWorkspaces` está permitido en Stripe, así que el webhook debe manejarlo: los workspaces excedentes (los más recientes primero) pasan a **read-only** (`Workspace.frozenAt`), sin borrar datos; banner "workspace congelado — amplía tu plan o elimina otro".
   - Cancelación/impago: `past_due` con gracia de N días → luego todos menos el workspace default quedan congelados y el default degrada a límites FREE.
   - Eliminar un workspace NO baja la quantity automáticamente (decisión explícita del OWNER desde billing, para evitar downgrades accidentales); mostrar aviso "estás pagando N y usando N-1".
6. **Checkout y portal:** `POST /api/billing/checkout` (Stripe Checkout, mode=subscription, `line_items: [{ price: tier, quantity }]`) y `POST /api/billing/portal` (Customer Portal para cambio de tier, método de pago y cancelación). Un solo customer por `User`.
7. **Webhook Stripe:** `POST /api/webhooks/stripe` con verificación de firma sobre **raw body** (mismo patrón que PR #19) e **idempotencia** por `event.id` persistido. Eventos: `checkout.session.completed`, `customer.subscription.updated` (incluye cambios de quantity), `customer.subscription.deleted`, `invoice.payment_failed`.
8. **Gating:** `PlanService.getEntitlements(userId)` → `{ tier, quantity, maxAccountsPerWorkspace, quotas }` consumido por: `workspaces.create` (PR #32), `integrations.connect` (cuentas por workspace) y los guards de cuota de PR #28 (que cuentan por `workspaceId` y resuelven el tier vía el OWNER del workspace).
9. **UI:** `/settings/billing` con tier, quantity contratada vs workspaces en uso, botón "añadir workspace", acceso al Portal, y estado de pago.

**Cambios sobre PRs existentes (modificar, no duplicar):**
- **PR #32:** su punto 8 queda resuelto: `maxWorkspaces = subscription.quantity` del OWNER; `maxAccountsPerWorkspace` = del tier. El flujo de creación de workspace integra el paso 4 de este PR. Añadir `Workspace.frozenAt DateTime?` (congelamiento del punto 5).
- **PR #28:** las cuotas por workspace resuelven su tier a través del OWNER del workspace (`WorkspaceMember.role = OWNER → Subscription`).
- **PR #13:** los endpoints de billing son personales (`@CurrentUser()`), no de workspace — excepción documentada al criterio de isolación.

**Archivos a modificar:**
- `libs/prisma/prisma/schema.prisma` (`Subscription`, `Workspace.frozenAt`, tabla `StripeEvent` para idempotencia + migración)
- Nuevo `apps/api/src/modules/billing/*` (billing.service, billing.controller, stripe.webhook.controller, plan.service)
- `apps/api/src/modules/workspaces/workspaces.service.ts` (check de quantity en create; estados frozen)
- `apps/web/src/app/settings/billing/page.tsx` (nueva), modal de upsell en el flujo de crear workspace
- `apps/api/src/main.ts` (raw body para `/api/webhooks/stripe`), env (`STRIPE_SECRET_KEY`, `STRIPE_WEBHOOK_SECRET`, price IDs por tier)

**Criterios de aceptación:**
- [ ] Checkout STARTER/PRO con quantity N → webhook → `Subscription` activa con `quantity = N`
- [ ] Crear workspace nº N+1 → 402 con upsell; aceptar → quantity N+1 en Stripe (prorrateado) → creación completa sin pasos manuales
- [ ] Con `BILLING_AUTO_QUANTITY_UPGRADE=false`, la creación queda bloqueada con enlace a billing
- [ ] Bajar quantity por debajo de los workspaces en uso congela los excedentes (read-only, sin pérdida de datos)
- [ ] Impago → gracia → congelamiento; reactivación del pago descongela
- [ ] Firma del webhook verificada con raw body; reprocesar el mismo `event.id` no duplica efectos
- [ ] FREE: 1 workspace y límites reducidos sin tocar Stripe

**Notas técnicas:**
Fuente de verdad de quantity = Stripe; nunca incrementar el espejo local sin confirmación del webhook (el flujo de la UI espera el webhook o re-consulta `GET /api/billing` con un pequeño poll). Usar Checkout + Portal (no Elements) para minimizar superficie PCI. Guardar `StripeEvent { id, type, processedAt }`. El precio unitario por tier vive en Stripe (price IDs en env); los límites (`maxAccountsPerWorkspace`, cuotas) en config del repo.
