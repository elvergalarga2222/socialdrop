# PR #08 — 🔴 Configurar retención de jobs BullMQ y política de Redis

**Área:** Performance / Infraestructura
**Prioridad:** 🔴 Crítico
**Esfuerzo estimado:** S (< 2h)

**Problema:**
Ningún `.add()` define `removeOnComplete/removeOnFail`; Redis corre con `allkeys-lru` + `maxmemory 256mb` (docker-compose), por lo que puede **evictar claves de BullMQ y perder publicaciones programadas** bajo presión de memoria. Sin límite, los jobs completados crecen sin fin.

**Solución propuesta:**
1. `defaultJobOptions: { removeOnComplete: 1000, removeOnFail: 5000, attempts: 3, backoff: { type: 'exponential', delay: 30000 } }` al registrar la cola (`BullModule.registerQueue`).
2. Mover BullMQ a una DB/instancia de Redis dedicada con `noeviction`; dejar `allkeys-lru` solo para caché.

**Archivos a modificar:**
- `apps/api/src/modules/posts/posts.module.ts` (registro de la queue)
- `docker-compose.yml` (política de Redis / segunda DB)

**Criterios de aceptación:**
- [ ] La cola BullMQ vive en un Redis/DB con `noeviction`
- [ ] Jobs completados/fallidos se podan automáticamente
- [ ] `attempts` + `backoff` aplican por defecto a todos los jobs
