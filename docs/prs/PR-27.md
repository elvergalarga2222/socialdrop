# PR #27 — 🔴 Backups automáticos de PostgreSQL

**Área:** Infraestructura
**Prioridad:** 🔴 Crítico
**Esfuerzo estimado:** S (< 2h)

**Problema:**
No existe **ninguna** estrategia de backup de Postgres en el repo (ni cron, ni pg_dump, ni volumen externo). Un fallo del VPS Contabo o una corrupción = pérdida total de datos de usuarios. Punto ciego más peligroso de la infraestructura actual.

**Solución propuesta:**
1. Script `scripts/backup-db.sh`: `pg_dump -Fc` diario, comprimido, subido a R2/S3/B2 (off-site) con retención de 30 días.
2. Cron en el host o servicio `backup` en docker-compose (imagen `postgres` + rclone/aws-cli).
3. Documentar y **probar** el procedimiento de restore.
4. Alertar si el backup falla (integrar con PR #23).

**Archivos a modificar:**
- Nuevo `scripts/backup-db.sh`
- `docker-compose.yml` (o crontab del host)
- `README.md`/docs (procedimiento de restore)

**Criterios de aceptación:**
- [ ] Dump diario off-site verificado (existe el archivo del día)
- [ ] Restore probado y documentado
- [ ] Fallo de backup genera alerta
