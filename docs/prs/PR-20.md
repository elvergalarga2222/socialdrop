# PR #20 — Throttling + backoff para Graph API y versión unificada

**Área:** Seguridad / Performance
**Prioridad:** 🟡 Importante
**Esfuerzo estimado:** M (2-8h)

**Problema:**
Sin throttling ni backoff ante rate limits de Meta Graph (`code` 4/17/32/613): un usuario de alto volumen puede agotar el rate limit y tumbar el sync de todos. Además las versiones de la API están dispersas: `InstagramProvider` usa `v18.0`, `MetricsService` y `BrainService` usan `v19.0` — riesgo de deprecación desincronizada.

**Solución propuesta:**
1. Centralizar `GRAPH_API_VERSION` en config y usarla en providers/metrics/brain.
2. Wrapper `graphFetch` con detección de códigos de rate limit → backoff exponencial con reintentos limitados.
3. Espaciar `syncAll` entre usuarios (escalonar el cron cuando haya multiusuario).

**Archivos a modificar:**
- `libs/integrations/src/providers/instagram.provider.ts`, `facebook.provider.ts`
- `apps/api/src/modules/metrics/metrics.service.ts`
- `apps/api/src/modules/brain/brain.service.ts`

**Criterios de aceptación:**
- [ ] Una sola constante de versión Graph en todo el código
- [ ] Errores de rate limit reintentan con backoff, no fallan duro
