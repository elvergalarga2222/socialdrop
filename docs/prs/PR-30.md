# PR #30 — 🔴 Scopes de Meta faltantes + App Review para DMs (bloquea Flow Builder e inbox en producción)

**Área:** Seguridad / Compliance
**Prioridad:** 🔴 Crítico — sin esto, Flow Builder, inbox y secuencias NO funcionan en producción con usuarios reales
**Esfuerzo estimado:** M (2-8h de código) + proceso de App Review de Meta (semanas de calendario — iniciar YA)

**Problema:**
`InstagramProvider.generateAuthUrl` solo pide `instagram_basic, instagram_content_publish, pages_show_list`. Ninguno de esos scopes permite **enviar DMs ni responder comentarios**. El Flow Builder (`SEND_DM`/`SEND_COMMENT`), la bandeja (`inbox.sendReply`) y las secuencias drip dependen de permisos que la app de Meta no tiene concedidos:

- `instagram_manage_messages` (DMs de Instagram)
- `pages_messaging` (DMs de Messenger/página)
- `instagram_manage_comments` (responder comentarios IG)
- `pages_manage_engagement` / `pages_read_engagement` (comentarios de página FB)

Hoy funciona solo porque se usa un token de env (`FACEBOOK_ACCESS_TOKEN`) generado a mano con permisos de admin/desarrollador. Con la app de Meta en modo **Live** y usuarios reales conectando sus cuentas vía OAuth, esas llamadas devolverán error de permisos: **toda la automatización de mensajería muere en producción**. Además, esos scopes requieren **App Review de Meta** (demostración en video, caso de uso, política de privacidad) — es un trámite de semanas que bloquea el calendario de lanzamiento tanto como cualquier bloqueante de código.

**Solución propuesta:**
1. **Código:** añadir los scopes de mensajería/comentarios a `generateAuthUrl` de Instagram y Facebook, condicionados por feature flag `ENABLE_MESSAGING_SCOPES` (para no romper el OAuth actual mientras la review está pendiente). Documentar en el flujo de conexión qué features requieren qué scopes.
2. **Manejo de permiso denegado:** si el usuario conecta sin conceder los scopes de mensajería (o la review no está aprobada), las features de automatización deben mostrarse como "requiere reconectar con permisos de mensajería" en la UI — no fallar en silencio (los `fetch` de `sendDM` hoy ni comprueban `res.ok`, ver PR #17).
3. **Proceso (no-código, iniciar en paralelo HOY):** preparar y enviar el App Review de Meta: screencast del flujo de DM automation, descripción del caso de uso, URLs de privacidad/borrado de datos (ya existen `/privacy`, `/privacy/deletion`), business verification si Meta la exige.
4. **Verificación:** test de integración que valide que el token obtenido por OAuth incluye los scopes esperados (`/debug_token` de Graph API) y panel en `/integrations` mostrando los permisos concedidos por cuenta.

**Archivos a modificar:**
- `libs/integrations/src/providers/instagram.provider.ts` (scopes en `generateAuthUrl`)
- `libs/integrations/src/providers/facebook.provider.ts` (ídem)
- `apps/api/src/modules/integrations/integrations.controller.ts` (exponer scopes concedidos vía `/debug_token`)
- `apps/web/src/app/integrations/page.tsx`, `flows/page.tsx`, `inbox/page.tsx` (estados "permiso faltante")
- `docs/` (checklist del App Review)

**Criterios de aceptación:**
- [ ] El OAuth solicita los scopes de mensajería/comentarios (tras flag/aprobación)
- [ ] `/integrations` muestra qué permisos tiene cada cuenta conectada
- [ ] Flow Builder/inbox/secuencias muestran aviso accionable si faltan permisos (no fallo silencioso)
- [ ] Checklist de App Review documentado y solicitud enviada a Meta
- [ ] Test que verifica los scopes del token vía `/debug_token`

**Notas técnicas:**
Este PR se movió de "nota de compliance" a bloqueante formal porque su timeline es el más largo del lanzamiento: el código es M, pero la App Review de Meta tarda semanas y sin ella las features de automatización — de las más vendibles del producto — no existen en producción. Coordinar con PR #17 (tokens por integración) y PR #19 (firma de webhooks): Meta revisará también que el webhook esté correctamente asegurado. La extensión Chrome de scraping debe mantenerse **fuera** del material del App Review (riesgo de rechazo de toda la app).
